        语法和语义

        什么是语法

        语法 (syntax) 是指代码文本的规范，只有符合语法的代码才可能被编译器/解释器处理。

        例如，a = 1; 是C语言的赋值语法，而 a := 1 在C语言中不合法。

        其中，一个重要的概念是语法结构，它通常以语法树 (syntax tree) 的形式展现:



        语法和BNF

        语法规则通常使用BNF文法定义。下面是四则运算的语法规则：

        e : addsub

addsub : addsub '+' muldiv
       | addsub '-' muldiv
       | muldiv

muldiv : muldiv '*' atom
       | muldiv '/' atom
       | atom

atom : '(' e ')' | NUMBER

        问题:

        1. "(1 + 1) + 1 "是否是合法的四则运算语言文本？

        2. "(1 / 2) + 1 "是否是合法的四则运算语言文本？

        3. "1.2"是否是合法的四则运算语言文本？

        4. 如果"1 + 1.2"对应语法树是Add(Int(1), Float(1.2)) ，

          问："1 + 1.2 * 2 "的语法树是？

        抽象语法树和具体语法树

        根据BNF文法，我们将语言的文本解析为语法树——文本在语法规则下的结构化表示。

        常用的语法树有两种：

        1. 具体语法树 (CST)： 除开结构化语法外，事无巨细地记录源代码信息。CST可以无损地转回源代码。
        2. 抽象语法树：只考虑后续工作需要的语法结构，抛弃不需要的信息，如空白符，分隔符，缩进等。

        举例子，给定一个语句 a = 1; , 通常类似于

        具体语法树:

        抽象语法树:

        		 
        Statement(
    kind = "assign",
    children = [
        Variable("a"),
        Token("="),
        IntLiteral(1),
        Token(";")
    ]
)
        		                               
        Assignment(
    target=Variable("a"),
    value=IntLiteral(1)
)
        上述的语法树仍然只是简化的形式（略去了行列号等），但可以看到抽象语法树的信息比具体语法树更少也更加紧凑。 在包括编译、解释和各式代码分析的大部分任务中，只会用到抽象语法树，而具体语法树没有用处。这也意味着语言的具体语法通常不影响语言的运行。 一门编程语言可以拥有多种具体语法，只要他们的语法结构和语义相同，他们之间的差异仍然是表面的，例如 f(x, y, z) 和 call f x, y, z 。

        重要的语法结构：抽象语法树

        抽象语法树是编译器/解释器工作的基础（通常是编译器/解释器的输入）。

        抽象语法树一般表示为代数数据类型 (ADT, Algebraic Data Type)，语法树常常被分为多个种类(sort)，绝大多数编程语言有以下的sort划分：

        • 表达式: 例如整数字面量表达式，函数调用表达式，二元运算表达式等。

        • 语句：例如赋值语句，if语句等。

        • 定义：例如数据类型定义，函数定义等。

        由上可见，每一个sort都可能继续划分。每一个sort可能有一到多个构造器 (constructor)。例如, 表达式sort有整数字面量、函数调用表达式和二元运算表达式这些constructor：

        Expr = IntLiteral(value: int)
     | FunctionCall(func: Expr, args: list<Expr>)
     | BinaryExpr(left: Expr, op: BinaryOperator, right: Expr)
     | ...

        上面定义sort的语言叫抽象语法定义语言(ASDL, Abstract Syntax Definition Language)。

        我们会频繁使用ASDL、BNF这些语言来提高沟通效率。

        以下是四则运算的语法树：

        ArithExpr = Add(left: ArithExpr, right: ArithExpr)
          | Sub(left: ArithExpr, right: ArithExpr)
          | Mul(left: ArithExpr, right: ArithExpr)
          | Div(left: ArithExpr, right: ArithExpr)
          | Number(value: number)

        什么是语义

        语义是对语法结构的解释。

        解释方式是多样的。打印语法树是其中一种解释。一般来说，我们关注的解释方式是代码执行 (evaluation) 。

        可以用如下框架表示evaluation的语义：

        语法结构X的执行结果是 r ⟺ X⇝r.

        这里的 X 是一个语法结构，如果对应编程概念，它是host language (实现编译器、解释器所使用的语言) 中的一颗语法树X的严格定义 ；

        这里的 r 是一个值，如果对应编程概念，它是target language中的一个值。和上面的X一样，用编程语言比喻这里的值并不精准。r的严格定义

        上面这个框架用来表达四则运算语言的语义绰绰有余：

        Add(Number(2), Number(2.0))⇝4.0

        在下文中，我们说语法结构时，都特指抽象语法树。除非表述无法推广到任意语法结构。

        语义和相继式

        合法的语法结构是不能枚举的，我们不可能写下：

        Add(1,Add(1,Add(⋯,Add(Number(1),Number(1))))⇝2992

        注意到语法结构的定义是递归的，我们也可以定义出一套递归的、有限的规则，为合法语法结构的全体描述语义。为了做到这一点，我们介绍相继式：

        c1c2c3c4⟺ 在条件c1c2c3都成立的情况下，c4成立。


        我们定义一些元符号 (可以大致理解为宿主语言中的变量)：

        i,i1,i2⋯in∈实数X,Y∈四则运算语言的语法结构全集f(i1,i2),f∈{+,−,*,/}分别表示表示计算两个实数的加/减/乘/除

        借助这些元符号，我们定义四则运算语言的完整语义(乘法，除法，减法被省略)：

        X⇝i1Y⇝i2Add(X,Y)⇝+(i1,i2)Number(i)⇝i Staging error

        带状态的语义

        在Java和C++语言中心，我们可以修改一个对象的字段。这是一个副作用，因为它们会引起某些函数对相同参数的调用返回不同的值。

        static struct MyTypeRef *val = { ... };

        int some_func(int factor)
{
    return val -> a * factor;
}
        

        int main()
{
    int result1, result2;
    result1 = some_func(2.0);
    val -> a = 1; // 对象val的字段a被修改
    result2 = some_func(2.0);
    // result1 != result2, 副作用！
    return 0;
}

        我们方才介绍的 X⇝r 不足以描述这个语义。我们原先只考虑了返回值，但一些语言中，语法结构涉及上面提到的副作用。

        副作用的本质是函数调用被隐式地增加了一个参数：全局状态参数。当我们写下f(arg) 的时候，我们实际上写下了f(GLOBAL_STATE, arg) 。

        因为程序的行为受全局状态影响，而语义应该完全决定程序行为，所以带有副作用的编程语言，其语义必须描述全局状态。在这种情况下，描述语义的框架通常是：

        (S,X)⟶(r,S)

        这是非常痛苦的。



        在本文之后的部分，我们的表述将会从理论上的“不精准”转变为“不正确”——这就是说，虽然我们仍使用(S,X)⟶(r,S)来描述一部分状态，但我们将忽略一些特殊的、我们不关心的副作用，例如IO等。例如，当我们的语义处理打印语句时，我们认为这里出现了一个函数调用，它实际做了打印，但我们的语义将其视为一个空返回值的纯函数。

        所幸的是，这种“不正确”丝毫不影响我们精准地理解程序语言的语义，它是一种折衷。如果你真的需要“更正确”的框架描述来描述具有副作用的语言的语义，你可能需要接触Haskell的副作用观点，它巧妙地将副作用表示为状态初值S0上发生的一系列变换： Sn=effn(⋯,eff2(eff1(S0)))。

        既然已经“不正确”了，便不妨追求更加刺激的语境。我们直言：在本文以下部分，语法结构是host language中的数据结构，而状态S以及返回值r是target language中的数据结构。为了让表述更加方便，我们让host language等于target language，这样一来，前文中Number(n)⇝n的含义就非常明确。

        一个最简单的解释器

        我们接下来讨论一个简单的语言，扩展的Λ演算语言 (LC/Lambda Calculus)。

        LC的语法结构和语义

        我们先用ASDL定义LC语法结构：

        LC = Call(func: LC, args: LC)
   | Var(varname: str)
   | Func(argname: str, body: LC)

        可以看到LC的语法结构简单得不得了，只有三种结构，分别是函数调用、读取变量和定义匿名函数。

        LC的语义也很简单，只是我们不会像教材一样介绍LC的约归 (reduction) 规则，因为介绍这个基本上没有屁用。之前说过，我们有host language和target language (并且在本文中，两者是一个语言)，我们要立刻将LC用一个实际的语言表达，以建立直觉。

        让我们选择 Python = target language = host language。本文之后提到target language，若非特殊说明，都指Python。

        为了给出语义，我们还需要定义一些辅助物件。

        我们需要一个状态 S，它表示LC作用域，一个从字符串到target language值的映射，可以被理解为target language中的字典:

        S=(a1↦r1)(a2↦r2)⋯(an↦rn)

        其中，a1,⋯,an是target language中的字符串。

        LC语义如下，其中 r,rf,r1,⋯rn 是我们target language中的值 (这样的值如果是函数，则是一个数学上的纯函数)：

        (a1↦v1)⋯(an↦vn),Var(ak)⟶vk,(a1↦v1)⋯(an↦vn)S,F⟶rf,S′S′,X⟶r1,S″(r2,S″′)=rf(S″,r1)S,Call(F,X)⟶r2,S″′∀S∀r((a↦r)S,X⟶r′,S′rf(S,r)=(r′,S))S*,Func(a,X)⟶rf,S*

        对Func的语义稍作解释：

        因为我们已经指定target language且target language等于host language，语法结构是该语言中的一个数据结构，返回值和状态也都是该语言中的数据结构，因此语义 S,X⟶r,S 可视为该语言中的一个函数。

        Func的语义解释中，函数rf的定义是由 ⟶ 导出的：

        在当前状态S*下定义一个纯函数rf，对于任意一个输入参数r以及一个调用者状态S，纯函数rf(S,r)的返回值是(r′,S′)，其中r′来自以下公式的推演：

        (a↦r1)S,X⟶r′,S′

        LC扩展、原语函数和解释器实现

        如果不扩展LC语言，它基本没用。这个局面在我们将语法结构做如下扩展时发生逆转。

        LC = Call(func: LC, args: LC)
   | Var(varname: str)
   | Func(argname: str, body: LC)
   | BoolVal(value: bool)
   | NumberVal(value: number)
   | StringVal(value: str)

        习题 尝试给出BoolVal, NumberVal 和StringVal 的语义？

        只做语法结构的扩展，LC仍然没有用处。因为我们甚至不能将数字做加法，不能将字符串做拼接。

        其实，即使LC语言不做上述扩展，也有一套自己的办法实现自然数、整数乃至列表、字符串，以及加法减法函数等等。但因为这些模拟没有什么用，如果不对PL感兴趣，但凡看一秒你都亏麻了。

        实用的路线是引入原语函数 (intrinsic function) 的概念。著名的实用派函数式语言Idris，其编译产物可以视为一种AST，这个AST会预设一些特殊名字的函数的存在，例如，Idris语言没有定义函数op_str_concat ，但任何字符串拼接操作最后都会调用到它。像op_str_concat 这样的函数被称为原语函数，它被预设存在，以提供语言最基本的、不可划分的操作。

        对于练习来说，我们可以任意预设原语函数的存在，最关键的是掌握思路。

        现在，让我们根据前面给出的LC语义，在Python语言 (建议Python 3.10以上) 中实现扩展LC的解释器。

import typing

arg: LC
        

        @dataclass(frozen=True)

name: str

        LC的语法解析

        欢迎来到编译器最平凡的部分，语法解析 (parsing)。语法解析对应的具体技术有parser generator和parser combinator等。

        解析器需要更好的报错和更强的表达力，但对极限性能的需求通常相当虚拟。使用ANTLR4 (一个相当实用的parser生成器框架)，按照lua规范实现一个lua解析器，这样的解析器中会存在相当多的歧义，多少会导致性能问题。但在实践中，上面这样“低质量”的lua解析器一秒钟能解析接近10万行来自real world的工业代码，这个性能完全足够让IDE分析工具在你每一次落键的瞬间重新解析、分析当前文件并更新补全信息。

        因此，大胆地去使用那些表现力足够强、报错足够好的解析器生成器！Tree-sitter，ANTLR4, Python Lark等等等。但要小心那些LL(1)/LR(1)的解析器，如果你不算parser专家，或者不想浪费生命去纠结为什么你的文法不工作。

        让我们使用Python的Lark。它的性能非常差，但是非常实用，被大量很重要的Python开源项目使用。

        解析器的工作是从源代码文本中提取语法结构——特指抽象语法树。

        下面是一个例子：

        start : lc

        lc    : call
      | atom
        

        call   : call atom
       | atom
        

        atom  : CNAME
      | func
      | number
      | boolean
      | string
      | "(" lc ")"
        

        string : ESCAPED_STRING
number : NUMBER
boolean: "true" | "false"
var    : CNAME
func   : "func" "(" CNAME ")" "{" lc "}"
        

        %import common.WS               // 这是空白符token
%import common.ESCAPED_STRING   // 这是双引号字符串token
%import common.CNAME            // 这是C标志服token
%import common.NUMBER           // 这是数字token
%ignore WS                      // 忽略所有空白符

        这个文法被一行代码的Lark编译工具处理后，可以生成一个LC语言的解析器：json_parser全流程。这个解析器输出的是具体语法树 (CST)，你可能需要手动转换CST到AST。

        出于可读性和可维护性的考虑，CST到AST的转换最好在文法中进行 (详见语法制导，或semantic actions)。Python的发展生动地说明了这点，虽然我们仍常常听到一些相反意见。对此，我的建议是努力地使用语法制导，但绝不在文法里处理引入除开构造AST以外的逻辑！

        // 一个好的使用semantic action的例子
lc_atom : CNAME   { Var(token_to_string($1)) }
        | "true"  { BoolVal(True) }
        | "false" { BoolVal(False) }
        | ...

        更丰富的语言及语言支持

        虽然我们已经完成一个有模有样的解释器，但它目前仍很不实用。

        该如何赋值呢？我们甚至不能使用x = 1 这样的代码。

        此外，函数体也只能支持“一句话”，而不能支持多行，就像下面这样：

        func (x)
{
    add x x
}

        要支持这些操作，我们引入两种新的语法结构：

        • 赋值

          • 语法结构：Block(body: list<LC>)

          • 语义： S,X⟶r,S′S,Assign(a,X)⟶r,(a↦r)S′

        • 语句块

          • 语法结构：Assign(name: str, value: LC)

          • 语义： S0,X1⟶r1,S1S1,X2⟶r2,S2⋯Sn−1,Xn⟶rn,SnS0,Block(X1X2⋯Xn)⟶rn,Sn

        习题 尝试在lc1.lark中为Assign和Block加入合适的语法规则，以支持下面的代码：

        func (x)

        描述语义时，我们使用了 S,X⟶r,S′的表述，编译可以视为对这个过程的一个拆解。

        我们考虑上图的如下流程：

        1. 语法结构到字节码。将其用X⟶compileI1I2⋯In的形式描述。
        2. 字节码到运行时值，即虚拟机解释。将其用S,I⟶VMr,S的形式描述。
